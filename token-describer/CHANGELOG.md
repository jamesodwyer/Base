# Token Describer Changelog

## v2.0.0 — 2026-04-13

Major skill rewrite focused on reducing bloat, closing pattern gaps, and improving validation quality.

### SKILL.md rewrite (465 lines -> ~170 lines)

**What changed**: Collapsed 12 procedural steps into 7 concise sections. Removed git workflow boilerplate, made documentation generation optional, broadened the trigger description.

**Why**: The original SKILL.md read like a step-by-step runbook, which is unnecessary overhead for an LLM — Claude already knows how to do git branching, committing, and PR creation. The verbose steps were consuming context window space without adding value. The 6 documentation files generated by default (TOKEN-DESCRIPTIONS.md, TOKEN-USAGE.md, etc.) added noise most runs didn't need. The trigger description was too narrow, missing common phrasings like "describe these tokens" or "$description fields".

### New token patterns in generate_descriptions.py

**What changed**: Added pattern matching for:
- `color.selected.control.*` — selectionControl fill/border/select states
- `stepper.color.*` — container, counter, and interactive button tokens
- `badge.spacing.*` — padding, gap, minHeight
- `selectionControl.spacing.*` — gap, validationGap, errorGap

**Why**: These components were added to the token files after the original skill was written. The generator was returning `no_match` for them, meaning they'd get no description on a fresh run. The stepper alone has 25 tokens that were invisible to the generator.

### Fixed color.disabled.c description

**What changed**: `color.disabled.c` was described as "Disabled border. Tertiary disabled." — changed to "Disabled checked fill. Selected controls."

**Why**: Per the project's documented conventions (CLAUDE.md memory and stepper token decisions), `disabled.c` is the heaviest disabled tier, used for selected/checked fills (e.g. a toggled checkbox when disabled). It was never for borders — `disabled.b` handles borders. The old description would actively mislead consumers into using the wrong token.

### Ghost vs tertiary button consistency

**What changed**: Split the combined "Tertiary (Ghost Buttons)" section in description-templates.md into two separate sections: "Ghost Buttons" and "Tertiary Buttons". Aligned the generator's `BUTTON_TYPES` mapping to treat ghost and tertiary as distinct variants.

**Why**: The templates called tertiary tokens "ghost buttons", but the generator had separate `ghost` and `tertiary` variants with different descriptions. In the actual token set, `interactive.ghost.*` and `interactive.tertiary.*` are distinct groups — ghost is minimal/transparent, tertiary is low-emphasis with a border. Conflating them in the templates created confusion about which description pattern to apply.

### Tighter validation component keyword check

**What changed**: Removed `"text"` and `"surface"` from the component mention keyword list. Added specific component names (stepper, toggle, checkbox, radio, switch, dropdown, popover, menu, accordion, slider, counter, control) and a separate context keyword list (heading, body, caption, label, CTA, action, overlay, elevation, shadow, canvas).

**Why**: The old check included words so broad that nearly any description would pass. A token described as "Use for primary text. Main content." would pass the component check via the word "text", even though it says nothing about which components use it. The two-tier approach (specific components + usage contexts) is more discriminating while still accepting descriptions that mention where a token is used rather than which component.

### Duplicate description detection

**What changed**: The validator now tracks all descriptions within a file and reports when multiple non-core tokens share the same description text. Core tokens are excluded since duplicates there are expected (e.g. multiple dimension tokens described as "Medium spacing.").

**Why**: The generator produces identical descriptions for some adjacent tokens (e.g. dimensions 300/350/400 all got "Medium spacing."). For core tokens this is acceptable — the scale position is in the token name. But for semantic or component tokens, identical descriptions indicate the generator is being too generic and the descriptions need manual refinement.

### Dry-run mode

**What changed**: Added `--dry-run` flag to `generate_descriptions.py`. Shows what would change (added/updated/kept counts) without modifying any files. Skips writing the run manifest.

**Why**: The generator writes directly to token files, which means running it to "see what would happen" required a git stash/restore cycle. A dry-run flag is standard practice for any tool that modifies files and eliminates the risk of accidental overwrites during review.

### Batch review workflow (in SKILL.md)

**What changed**: Replaced the one-by-one `AskUserQuestion` review loop with a confidence-tiered batch approach: auto-approve core/brand tokens, present semantic tokens in category groups, flag only genuinely ambiguous tokens individually.

**Why**: The original Step 7 would prompt the user individually for every token without Figma context. For a 558-token set where most lack Figma bindings, this means 100+ sequential interruptions. The batch approach lets users see related tokens side-by-side (all `input.*` together, all `feedback.*` together), spot inconsistencies across a group, and review at their own pace. The existing `batch-review-workflow.md` reference and `generate_review_report.py` / `apply_review.py` scripts already support this — the SKILL.md just wasn't pointing to them.
